w4afp8 代码逻辑与支持情况（SGLang）

本页面基于仓库内代码静态阅读整理，重点回答两件事：逻辑怎么走、当前支持到什么程度。

1) 一句话结论

  • w4afp8 是一种“混合量化”路径：普通 Linear 走 FP8，MoE 专家层走 W4A8（int4 weight + fp8 activation 的 kernel 路径）。
  • 该路径对硬件有明确门槛：配置中要求 compute capability >= 90（Hopper/SM90 级别）。
  • 与 ROCm 的兼容列表中 未包含 w4afp8，因此当前主要是 CUDA/Hopper 场景。

2) 入口与识别逻辑

2.1 启动参数入口

服务启动参数 --quantization 的可选值里包含 w4afp8，可显式指定。

2.2 自动识别（模型配置）

当模型目录下存在 hf_quant_config.json 且 quant_algo == "MIXED_PRECISION" 时，代码会自动映射为 {"quant_method": "w4afp8"}。

3) 核心执行逻辑（按层类型分流）

层类型 w4afp8 下采用的方法 说明
LinearBase Fp8LinearMethod 普通线性层沿用 FP8 线性实现；可通过 ignored_layers 跳过量化。
FusedMoE W4AFp8MoEMethod MoE 专家层创建 int8 容器存储压缩权重及 scale，并在加载后做 scale 交织和输入 scale 归一处理。

3.1 W4AFp8MoEMethod 关键步骤

  1. 创建专家参数：w13_weight、w2_weight（int8 张量容器）及对应 _scale_inv。
  2. 创建输入 scale：w13_input_scale、w2_input_scale。
  3. 预生成 grouped GEMM 所需 strides、offsets、problem size 缓冲。
  4. 加载后处理：对权重 scale 做“每 4 个一组交织（interleave）”，并把输入 scale 压成单一 max 值，便于 kernel 使用。

4) 支持情况（现状）

维度 状态 依据
CLI 可用 支持 --quantization choices 含 w4afp8。
自动识别 MIXED_PRECISION 支持 模型量化配置解析时会映射到 w4afp8。
最小算力 要求高 get_min_capability() 返回 90。
ROCm 当前不在支持列表 ROCm 支持量化方法白名单中未出现 w4afp8。
MoE kernel 形态 已实现且做了 shape 分支优化 SM90 cutlass grouped GEMM 路径对不同 (n,k,m) 选择不同 tile/cluster 配置。

5) 典型使用建议

  • 优先在 Hopper（SM90）GPU 上使用。
  • 若模型是 modelopt 的 MIXED_PRECISION 导出，可先不手工指定，验证自动识别是否命中 w4afp8。
  • 排障时优先确认：量化方法是否被覆盖、是否为 ROCm 环境、MoE 维度是否落在常见优化分支。

6) 可复制命令（示例）

python -m sglang.launch_server \
  --model <your_model_or_path> \
  --quantization w4afp8 \
  --dtype bfloat16
